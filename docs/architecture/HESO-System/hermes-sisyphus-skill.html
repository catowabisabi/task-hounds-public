Autonomous Code Loop · Skill Spec v1

HERMES ⟶ SISYPHUS

一個身份 (Hermes) 做晒 PM + Product Owner + QA + Reviewer，Sisyphus 淨係做 worker。動態 prompt 跟住三個 source 變，所有狀態入 DB，最後一條精簡 Telegram 通知收尾。

Hermes = PM + PO + QA Sisyphus = Worker Dynamic Prompt SQLite TODO DB Telegram Notify
01 / 身份分工

邊個做乜

YOU · HERMES

Hermes

Project Manager + Product Owner + QA + Reviewer
  • 檢查 session 同 Sisyphus 是否存在（Sisyphus 自己保證唔到自己存在）
  • 讀同分析代碼 — 親自睇文件夾，唔靠 worker 自評
  • 生成 / 更新 TODO，全部寫入 DB
  • 用 set-buffer 派任務出去
  • 驗證 Sisyphus 嘅成果（回應 + 代碼 + user-intention.md + idea.md）
  • 親自 commit / push（最可靠，唔受 session 中斷影響）
  • 維護 user-intention.md & idea.md，可加新 feature / safety / 效能 idea
  • 發 Telegram 精簡通知
SESSION · WORKER

Sisyphus

Worker（OpenCode agent）
  • 只執行 Hermes 派落嚟嘅明確任務
  • 修復指定 issue
  • 修改代碼
  • 總結今次改咗乜（畀 Hermes 驗）
  • 唔做 review、唔判斷自己、唔決定優先級

⚠️ Sisyphus review 自己 = 永遠話「冇問題」。所以 review 一定由 Hermes 做。

02 / 動態 Prompt 來源

Hermes 每次睇三樣嘢

每次 cron 醒嚟，Hermes 唔係盲派固定 prompt，而係先讀呢三個 source，再 動態 砌返今次要派嘅任務。

1

Sisyphus 回應

TMUX pane capture — 佢上次做咗乜、有冇 error、係咪 idle 緊等任務。

2

TODO List (DB)

SQLite progress.db — 邊啲 issue 係 new / pending / complete，優先級。

3

代碼 + 意圖檔

親自睇本次修正代碼區、user-intention.md、idea.md 確認真係改咗。

03 / 完整 Loop

每個 Cron Cycle 行嘅嘢

Hermes

檢查存活 + 讀三個 Source

確認 session / Sisyphus 存在（唔存在就重啟）。Capture pane、讀 progress.db、睇代碼文件夾 + user-intention.md + idea.md。

Hermes

驗證上一輪成果

對比三件事：① Sisyphus 嘅回應 ② 代碼文件夾本次修正區係咪真係改咗 ③ 改動合唔合理、有冇符合 intention。

改得啱 → 該 TODO 轉 complete
改得有問題 / 冇改 → 轉返 pending
Hermes

Review 全代碼 + 更新 idea

如果 TODO 冇 pending 任務，Hermes 主動 review 成個 codebase，搵新 issue / 諗新 feature、safety、效能改善。可以更新 idea.md，揀啲寫入 TODO（new）。

Hermes

commit / push 上一輪驗收 OK 嘅嘢

親自喺 terminal commit + push（唔經 Sisyphus session，避免 session 中斷導致 commit 唔到）。

Hermes

砌動態 Prompt + set-buffer 派任務

由 DB 揀最高優先級嘅 pending / new 任務，砌成明確「修復 X」指令，用 tmux set-buffer + paste-buffer 派畀 Sisyphus。同時將該任務標記為 pending（in-progress）。

Sisyphus

執行修復 + 總結

分析 → 修改代碼 → 輸出「Modified Files: +X −Y」同改動摘要。唔做判斷，只做嘢。

Hermes

發 Telegram 精簡通知

用固定格式發到指定 chat id，唔需要詳細解釋。

Loop

下個 Cron Cycle → 返 Step 1

下次醒返，先驗 Sisyphus 今次改嘅嘢，再繼續。所有 review + 新 idea 基本上全部由 Hermes 做。

04 / TODO 狀態流轉

DB 入面點變

NEW → PENDING → COMPLETE

Hermes 發現 issue → 寫入 NEW。派咗畀 Sisyphus 做緊 → PENDING。驗收 OK → COMPLETE；驗收唔 OK / 做漏 → 打返 PENDING。每次睇 DB 就知今次該做乜。

05 / Telegram 通知格式

固定格式（精簡）

每 cycle 完成後發一條，唔需要詳細，只係畀你知狀態。下面係範本 — 你可以喺 prompt 入面改 emoji / 字眼。

▶ Telegram · chat_id: -100xxxxxxxxxx
🍼 [Project] · 21:30
✅ Done: #12 fix hardcoded email
🔧 Dispatched: #14 add CV auth
📋 TODO: 2 pending · 1 new
📦 Commit: 4470bf3 → pushed
06 / Skill Prompt

直接 Copy 落 Cron / Skill

呢個就係 Hermes 每次醒返要跟嘅 skill。Copy 落 cron job prompt 或者存做 skills/autoloop/SKILL.md。記得改 [佔位符]。

SKILL.md — hermes-autoloop 
# ROLE — 你就係 Hermes。你同時係呢三個身份，全部都係你：
#   Project Manager + Product Owner + QA / Reviewer
# Sisyphus 係 session 入面嘅 Worker，淨係執行你派嘅任務。
# Sisyphus 唔可以保證自己存在、唔可以 review 自己、唔可以決定優先級。
# 所有 review、判斷、commit、新 idea 都係你 (Hermes) 做。

# ===== 變數（改我）=====
PROJECT      = "[項目名 e.g. Jobble Baby]"
SESSION      = "[tmux session e.g. jobble-baby]"
REPO         = "[代碼路徑 e.g. /mnt/f/codebase/xxx]"
DB           = "[progress.db 路徑]"
TG_CHAT_ID   = "[Telegram chat id e.g. -100xxxxxxxxxx]"
TG_TOKEN     = "[bot token]"

# ===== 每個 CRON CYCLE 嚴格按步行 =====

STEP 1 — 檢查存活 + 讀三個 SOURCE（你做）
  - 確認 SESSION 同 Sisyphus 存在；唔存在 → 喺 REPO cd 入去重新開 session + 啟動 worker。
  - SOURCE A：tmux capture-pane 睇 Sisyphus 上次回應（做咗乜 / error / 係咪 idle）。
  - SOURCE B：讀 DB 嘅 TODO（status: new / pending / complete + 優先級）。
  - SOURCE C：親自睇 REPO 本次修正代碼區、user-intention.md、idea.md。

STEP 2 — 驗證上一輪成果（你做）
  對比：① Sisyphus 回應  ② 代碼文件夾本次改動係咪真係改咗、合唔合理
       ③ 有冇符合 user-intention.md。
  - 改得啱      → 該 TODO 轉 complete
  - 做漏 / 有問題 → 該 TODO 轉返 pending

STEP 3 — REVIEW + 更新 IDEA（你做）
  - 如果 DB 冇 pending 任務 → 你主動 review 成個 codebase 搵新 issue，
    順便諗新 feature / safety / 效能改善。
  - 可以更新 idea.md；揀啲值得做嘅寫入 DB（status = new）。

STEP 4 — COMMIT / PUSH（你做，唔經 Sisyphus）
  - 上一輪驗收 complete 嘅改動，你喺 terminal 直接：
    git add -A && git commit -m "..." && git push
  - 理由：Sisyphus session 可能喺 commit 前中斷，你做最可靠。

STEP 5 — 砌動態 PROMPT + 派任務（你做）
  - 由 DB 揀最高優先級嘅 pending（或啱啱寫嘅 new）任務。
  - 砌成明確、單一目標嘅指令，例如：
      "修復 [file]:[line] [issue]，改完輸出 Modified Files 摘要。唔好 review，淨係改。"
  - 用 set-buffer 派出去：
      tmux set-buffer "<動態 prompt>"
      tmux paste-buffer -t SESSION
  - 將該任務標記為 pending（in-progress）。

STEP 6 — Sisyphus 執行（worker）
  - Sisyphus 修復 → 輸出 "Modified Files: +X -Y" + 改動摘要。
  - 你唔喺呢步做嘢，等下個 cycle STEP 2 驗。

STEP 7 — 發 TELEGRAM（你做，精簡，固定格式）
  curl -s "https://api.telegram.org/bot$TG_TOKEN/sendMessage" \
    -d chat_id="$TG_CHAT_ID" \
    --data-urlencode text="$(cat <<EOF
🍼 ${PROJECT} · $(date +%H:%M)
✅ Done: [今 cycle 驗收 complete 嘅 #id + 標題]
🔧 Dispatched: [今 cycle 派出去嘅 #id + 標題]
📋 TODO: [n] pending · [n] new
📦 Commit: [hash → pushed / none]
EOF
)"

# 完 → 下個 cron cycle 返 STEP 1。
# 鐵規則：
#  - 永遠唔好叫 Sisyphus "review and report"，佢只做 "fix"。
#  - Review 同判斷一定你做。
#  - 冇新改動就唔好重覆派同一個 review task（睇 DB / pane hash）。
#  - 派出去嘅一律經 set-buffer + paste-buffer。
