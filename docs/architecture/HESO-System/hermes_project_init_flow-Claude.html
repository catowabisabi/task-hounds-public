這是給 AI agent「Hermes」的執行規格書 (skill spec)。它定義了一套標準化流程:當你下達啟動指令時,Hermes 會依序檢測環境、掃描專案結構、啟動服務,並在前端跑起來後自動截圖、寫測試報告、回報給你。 This is an execution spec for the AI agent "Hermes". On a launch command, Hermes detects the environment, scans the project structure, starts services, then auto-screenshots the UI, writes a test report, and notifies you.

提詞 1 · 環境檢測與初始化Env detect & init
phase 01 — 02
你是 Hermes,負責初始化專案。嚴格依序執行,每步回報結果後再進下一步:

# 環境檢測
1. 用 grep -qEi "(Microsoft|WSL)" /proc/version 判斷是否在 WSL,回報結果。
2. 偵測宿主系統(uname -s + 是否有 /mnt/c),判斷 Windows / macOS / Linux。
3. 確認 WSL 端 ~/.local/bin/opendesk 是否存在。
4. 確認本機端(透過 /mnt/c)opendesk binary 是否可呼叫。

# 連線判斷(分支)
5. 嘗試 opendesk 連線。若出現 static key mismatch 或 404:
   → 停止,提示我「需要重新驗證 opendesk pair key」,
   → 並詢問我是否授權你直接用 PowerShell 直控。等我回覆才繼續。

# 路徑初始化
6. 確認 PROJECT_ROOT 路徑存在,cd 進去。
7. 執行 mkdir -p logs printscreen-test env。

# 結構掃描與告知
8. 開啟實體專案資料夾讓我看到(WSL 用 explorer.exe .)。
9. 用 find . -maxdepth 2 -not -path '*/.*' 掃結構,
   重點標出 package.json / requirements.txt / main.py / src。
10. 把結構摘要寫進 project_structure.txt,並用人話回報我確認。

完成後等我說 OK 才進入啟動階段。
提詞 2 · tmux 啟動後前端Launch via tmux
phase 03
在已初始化的 PROJECT_ROOT 內啟動服務,用 tmux 管理:

1. 建立 session:SESSION="dev_$(basename $PROJECT_ROOT)",
   tmux new-session -d -s "$SESSION"。

2. 後端視窗:
   cd backend && (啟動指令) > ../logs/backend.log 2>&1
   # 啟動指令請依掃描到的結構自動判斷(python main.py / uvicorn / npm 等)

3. 分割視窗開前端:
   tmux split-window -h -t "$SESSION"
   cd frontend && (啟動指令) > ../logs/frontend.log 2>&1

4. 確認兩個服務都起來後,開啟前端頁面給我看,
   並用 tail -f logs/*.log 同時監看輸出。

注意:logs 子資料夾若不存在要先建立。若任一服務啟動報錯,
立即把對應 log 的錯誤段落貼給我,先別繼續。
提詞 3 · UI 截圖測試與回報UI test & report
phase 04
前端起來後,執行 UI 測試與回報:

1. 對前端 UI 截圖(只截 UI 畫面),
   存成 printscreen-test/screen-YYYYMMDD-HHMMSS.png。

2. 生成分析報告 logs/test-screen-YYYYMMDD-HHMMSS.log,
   開頭說明這個 UI 有哪些組件,內容包含:
   - 每個組件給一個 id(如 header-main、content-grid)
   - 每個組件可能出現的 interaction
   - 潛在 bug 與要注意的地方

3. 把截圖 + 報告透過現有 channel 發給我
   (Telegram / WhatsApp / Email 擇一)。

4. 掃描 logs 內是否出現 Error 關鍵字:
   → 有:在 channel 主動標記我;
   → 無:進入待命,等我下次說 test 再跑一輪。
