This document is intended to serve as a GitHub-friendly specification for a reusable automation skill. It defines a safe and structured process for opening a project, validating the execution environment, launching services, capturing UI screenshots, generating structured logs, and sending reports back to the user.

Workflow Summary / 流程摘要

Short explanation of the process in repository-friendly bilingual format.

中文說明

呢個 Skill 主要用嚟處理「打開 project 並驗證 UI」呢類高頻任務。流程首先會確認專案路徑、 環境類型（WSL / Windows / macOS / Linux），再檢查 OpenDesk / OpenHand 是否可用。 如果 remote control 失敗，流程會 fallback 去 terminal 模式，並喺必要時要求用戶確認。

完成環境檢查後，系統會建立 logs/、printscreen-test/、env/，同時確保 .gitignore 已排除 logs、截圖、env、.db、.png、.jpg、.jpeg、.log、.env 等檔案。之後會掃描專案結構，判斷 backend / frontend 啟動方式，開兩個 terminal，將所有輸出寫入 logs。

當 frontend 成功啟動後，流程會打開 UI、截圖、分析畫面組件、建立帶有 component ID 的測試報告， 並經由既有 channel 發送俾用家。最後，agent 會提供整體狀態與下一步建議。

English Summary

This skill is designed for repeatable “open project and validate UI” tasks. It begins by confirming the project location and execution environment (WSL / Windows / macOS / Linux), then checks whether OpenDesk / OpenHand is available for remote interaction.

If remote control is not available, the workflow safely falls back to terminal-based execution and asks the user for confirmation when needed. It then prepares the working directories, updates .gitignore, scans the project structure, determines how backend and frontend should be launched, and records all output to timestamped log files.

Once the frontend UI is running, the workflow opens the page, captures a screenshot, analyzes the UI, writes a structured test log with component IDs and potential interactions/bugs, and sends the result to the user through an existing communication channel.

Detailed Stages / 詳細步驟

Expanded explanation of each stage for future implementation or skill authoring.

0

確認專案位置 / Confirm Project Path

接收 project 名稱或完整路徑，檢查 folder 是否存在，避免誤開錯誤專案。

  • Accept project name or full path
  • Resolve path in WSL / Windows / Unix-like systems
  • Stop early if the folder cannot be found
1

偵測執行環境 / Detect Environment

判斷目前是否在 WSL，並識別外層作業系統，建立 path mapping。

grep -qEi "(Microsoft|WSL)" /proc/version uname -s
2

檢查控制工具 / Check Control Tools

確認 OpenDesk / OpenHand 是否安裝及可用，檢查 pair / session 狀態。

  • Check local binary availability
  • Validate pair/session state
  • Handle static key mismatch safely
3

Fallback 決策 / Fallback Decision

如果 remote control 唔通，就用 terminal / PowerShell 模式；有需要時先問用戶確認。

  • Prefer OpenDesk / OpenHand first
  • Fallback to terminal-based launch
  • Ask user when direct control is required
4

建立資料夾 / Create Required Folders

確保 logs/、printscreen-test/、env/ 存在，方便輸出與測試資源集中管理。

mkdir -p logs printscreen-test env
5

更新 .gitignore / Update .gitignore

將暫存及測試輸出排除，保持 repo 整潔。

/logs/ /printscreen-test/ /env/ *.db *.jpg *.jpeg *.png *.log *.logs .env
6

打開專案資料夾 / Open Project Folder

喺打開之前先通知用家，避免背景操作令人混亂。

  • Windows: explorer.exe
  • macOS: open
  • Linux: xdg-open
7

掃描專案結構 / Scan Project Structure

檢查 frontend/backend 目錄、package.json、requirements.txt、main.py 等重要入口。

  • Detect framework and stack
  • Identify launch entry points
  • Write scan result to logs
8

判斷啟動方式 / Decide Launch Method

如果入口清晰就自動啟動；如果唔清晰就列出候選 command 俾用家確認。

  • Backend: Python / Node / Docker / etc.
  • Frontend: npm / pnpm / yarn / Vite / Next / React
9

開 Backend / Frontend / Launch Services

為 backend 同 frontend 各開一個 terminal，並寫入獨立 log 檔案。

backend > logs/backend-YYYYMMDD-HHMMSS.log 2>&1 frontend > logs/frontend-YYYYMMDD-HHMMSS.log 2>&1
10

打開 Frontend UI / Open Frontend UI

從 log 或預設 port 偵測 URL，然後打開瀏覽器。

  • localhost:3000
  • localhost:5173
  • localhost:8080
11

截圖 / Capture Screenshot

將 UI 畫面存入 printscreen-test/，只保留測試用途，不提交到 Git。

printscreen-test/screen-YYYYMMDD-HHMMSS.png
12–14

分析與回報 / Analyze and Report

建立 UI component list、component ID、interaction、潛在 bug，然後經 channel 傳俾用家。

logs/test-screen-YYYYMMDD-HHMMSS.log
15

錯誤處理 / Error Handling

包括 port conflict、static key mismatch、missing path、launch failure 等。

  • Do not kill processes blindly
  • Explain the issue clearly
  • Offer the single most useful next step
16–17

最後回報與下一步 / Final Report and Next Action

向用家回報 project folder、啟動狀態、frontend URL、screenshot 位置、log 檔案、以及建議下一步。

18

核心原則 / Core Principle

Verify → Inform → Execute → Log → Report → Protect

  • 先驗證環境，再做操作
  • 先通知用家，再打開或控制
  • 所有輸出都必須可追蹤
  • 測試資源不污染 Git repository
